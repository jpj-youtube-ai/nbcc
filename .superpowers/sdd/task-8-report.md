## Gmail reply bridge

`gmail-reply.js` is an ES module (`export function buildGmailReplyUrl`, `export function
formatReceived`), unit-tested by `test/unit/gmail-reply.test.js` (untouched). `app.js` is a
classic-script IIFE, so `admin.html` bridges the two with:

```html
<script type="module">
  import { buildGmailReplyUrl, formatReceived } from "/assets/js/gmail-reply.js";
  window.buildGmailReplyUrl = buildGmailReplyUrl;
  window.formatReceived = formatReceived;
</script>
```

placed after the two `defer` scripts in `<head>`. `app.js` calls `window.buildGmailReplyUrl(row)`
/ `window.formatReceived(...)` only from click handlers and render functions, after both scripts
have had a chance to execute.

## Honest-save form behaviour (assets/js/main.js)

- Validation-fail path unchanged (still clears `status`, focuses first invalid field, no fetch).
- On a valid submit: button disabled, status set to "Sending…", `POST /api/contact` is fired.
  - `res.ok` → success copy + `.form-status.is-success`, `form.reset()`, `clearErrors()`.
  - `!res.ok` or a rejected fetch → error copy + `.form-status.is-error`; form is **not** reset
    (message field, and everything else, keeps its typed value).
  - Button is always re-enabled in the trailing `.then()`.
- `typeof win.fetch !== "function"` still short-circuits before any status/button mutation
  (native no-JS POST handles it) — same guard as before.
